# 設計駆動開発ドキュメント

## 1. 目的

本ドキュメントは、許容領域シミュレータを設計駆動で開発するための作業基準を定義する。  
要件定義から詳細設計、実装、検証までを一貫した設計文書に基づいて進めることを目的とする。

## 2. 開発方針

本開発では、先に UI を作り始めるのではなく、次の順序で進める。

1. 要件定義を確定する
2. 数理モデルを設計する
3. データモデルを設計する
4. 判定アルゴリズムを設計する
5. 許容量探索アルゴリズムを設計する
6. UI 設計を行う
7. 実装する
8. テストと検証を行う

## 3. このプロジェクトで守る原則

### 3.1 判定と表示を分離する

UI 上の表示ロジックと、幾何計算・判定ロジックを分離する。

### 3.2 6DoF の入力値を直接判定に使わない

判定は必ず `評価点での変位` に変換して行う。

### 3.3 Pass / Fail だけで終わらせない

各軸の許容範囲、最悪点、残余マージンを設計上の主要出力とする。

### 3.4 不確かさを最初から組み込む

後付けではなく、設計段階から `U_identify`、`U_surrogate` などをモデルに含める。

### 3.5 最遠点 1 点で固定しない

実装初期は最遠点で始めてもよいが、設計は複数評価点対応を前提にする。

## 5. 開発ステップ

### Step 1: 問題定義

最初に次を固定する。

- 判定対象部位
- セットアップマージンの定義
- 評価点の与え方
- 不確かさの初期値

### Step 2: 数理モデル設計

先に数式を固定する。  
実装前に次を文章と数式で定義する。

- 並進の符号規約
- 回転の符号規約
- 回転の適用順序
- 回転中心
- 変位の算出方法

### Step 3: データ構造設計

次のデータを、コードに落としやすい形で定義する。

- `SetupState`
- `EvaluationPoint`
- `MarginProtocol`
- `UncertaintyModel`
- `PointResult`
- `AxisAllowance`
- `SimulationResult`

### Step 4: 判定ロジック設計

現在の 6DoF に対して、全評価点の判定結果を返すロジックを設計する。

### Step 5: 許容探索ロジック設計

各軸について、単独許容量と条件付き許容量を探索する方法を設計する。  
探索アルゴリズムは、まず二分探索を基本とする。

### Step 6: UI 設計

判定結果をどう見せるかを設計する。  
重要なのは、数値、表、スライダー、色分けが矛盾なく連動することである。

### Step 7: 実装

設計文書に従ってモジュール単位で実装する。

### Step 8: 検証

数式どおりに計算されているか、境界値で破綻しないかを確認する。

## 7. モジュール設計方針

### 7.1 geometry

責務:

- 6DoF から回転行列を作る
- 評価点変位を計算する
- 回転寄与を計算する

### 7.2 uncertainty

責務:

- 不確かさパラメータを保持する
- RSS 合成する
- 保守的変位を算出する

### 7.3 decision

責務:

- 各点・各軸の Pass / Fail を判定する
- 最悪点を抽出する
- 制約因子を抽出する

### 7.4 allowance

責務:

- 単独許容量を探索する
- 条件付き許容量を探索する
- 境界値を数値的に解く

### 7.5 app

責務:

- 入力を受け取る
- 計算結果を表示する
- 再計算をトリガする

## 8. 先に固定すべき設計論点

実装前に、次の設計論点を必ず確定する。

### 8.1 座標系

- Vertical / Long / Lateral の向き
- 単位
- 原点

### 8.2 回転仕様

- Rotation / Pitch / Roll の対応軸
- 符号
- 角度単位
- 適用順序

### 8.3 判定単位

- 軸別判定を主とするか
- 3D 判定を併記するか

### 8.4 不確かさの粒度

- 等方性にするか
- 軸別にするか
- 点別にするか

### 8.5 許容量の基準状態

- 単独許容量で他軸を 0 固定にするか
- 現在値固定にするか

## 9. 実装前レビュー項目

実装開始前に、次を確認する。

- 数式が要件と整合しているか
- 回転中心の扱いが明確か
- UI で表示する指標が定義済みか
- 許容量計算の探索範囲が定義済みか
- 境界値の扱いが明確か

## 10. テスト駆動ではなく設計駆動で重視する点

この開発では、単にテストを書く前に、`何を正しいとみなすか` を設計文書で固定することを優先する。  
特に次が重要である。

- 幾何学モデルの正しさ
- 符号規約の一貫性
- UI 表示と内部判定の一致
- 単独許容量と条件付き許容量の区別

## 14. 結論

本プロジェクトでは、`許容領域シミュレータ` を単なる UI 実装として始めるのではなく、`数理モデル`、`データ構造`、`判定仕様`、`許容量探索仕様` を先に定義してから実装へ進む。  
この順序により、放射線治療のセットアップマージンという幾何学的かつ安全性要求の高い問題に対して、一貫性のある開発を進めやすくなる。
